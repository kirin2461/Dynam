## Critical Issues (R14)

### 🔴 R14-C01: malloc Without NULL Check After Allocation

| Property | Value |
|----------|-------|
| **File** | `src/core/src/spoofer.cpp`, `network.cpp`, `dhcp_spoofer.cpp` |
| **Lines** | spoofer.cpp:549, network.cpp:90,131, dhcp_spoofer.cpp:161 |
| **Severity** | 🔴 **CRITICAL** (CWE-476: NULL Pointer Dereference) |
| **CVSS** | 7.5 (High) |
| **Status** | ⬜ Open |

**Vulnerable Code Pattern #1** (`spoofer.cpp:549`):
```cpp
// R10-M01: Dynamic buffer sizing for GetAdaptersAddresses
ULONG bufferSize = 15000;
PIP_ADAPTER_ADDRESSES addresses = nullptr;

for (int attempts = 0; attempts < 3; ++attempts) {
    addresses = (PIP_ADAPTER_ADDRESSES)malloc(bufferSize);
    if (!addresses) return false;  // ✅ NULL check present

    DWORD result = GetAdaptersAddresses(AF_UNSPEC, GAA_FLAG_INCLUDE_PREFIX, 
                                         nullptr, addresses, &bufferSize);
    if (result == NO_ERROR) {
        break;
    } else if (result == ERROR_BUFFER_OVERFLOW && attempts < 2) {
        free(addresses);  // ✅ Safe: addresses is non-NULL here
        bufferSize *= 2;
    } else {
        free(addresses);  // ✅ Safe: addresses is non-NULL here
        return false;
    }
}

if (!addresses) return false;  // ✅ Second NULL check
```

**Assessment**: ✅ **NOT VULNERABLE** — NULL checks are present. However, pattern is error-prone.

**Vulnerable Code Pattern #2** (`network.cpp:90`):
```cpp
std::vector<Network::InterfaceInfo> Network::get_interfaces() {
    PIP_ADAPTER_INFO adapter_info = nullptr;
    ULONG buf_len = 0;
    GetAdaptersInfo(adapter_info, &buf_len);  // Get required size
    adapter_info = (IP_ADAPTER_INFO*)malloc(buf_len);

    if (GetAdaptersInfo(adapter_info, &buf_len) == NO_ERROR) {
        // ... use adapter_info ...
    }
    free(adapter_info);  // ❌ CRITICAL: free(NULL) if malloc failed!
}
```

**Exploit Scenario**:
1. System under memory pressure
2. `malloc(buf_len)` returns NULL
3. Code continues to call `GetAdaptersInfo(adapter_info, ...)` with NULL
4. **NULL pointer dereference** in Windows API
5. Later: `free(NULL)` — technically safe per C standard, but indicates poor hygiene

**Root Cause**: No NULL check between `malloc()` and first use.

**Fix**:
```cpp
adapter_info = (IP_ADAPTER_INFO*)malloc(buf_len);
if (!adapter_info) {
    return interfaces;  // Return empty vector on allocation failure
}

if (GetAdaptersInfo(adapter_info, &buf_len) == NO_ERROR) {
    // ... use adapter_info ...
}
free(adapter_info);
```

**Also affects**:
- `network.cpp:131` — Same pattern
- `dhcp_spoofer.cpp:161` — Same pattern (but has NULL check in loop)

## High Priority Issues (R14)

### 🟠 R14-H01: malloc/free Without RAII Wrapper

| Property | Value |
|----------|-------|
| **File** | `spoofer.cpp`, `network.cpp`, `dhcp_spoofer.cpp` |
| **Lines** | Multiple |
| **Severity** | 🟠 **HIGH** (CWE-477: Obsolete Function) |
| **Status** | ⬜ Open |

**Issue**: Code uses C-style `malloc/free` instead of C++ RAII patterns.

**Example** (`spoofer.cpp:549-565`):
```cpp
PIP_ADAPTER_ADDRESSES addresses = (PIP_ADAPTER_ADDRESSES)malloc(bufferSize);
if (!addresses) return false;

// ... use addresses ...

free(addresses);  // Manual cleanup
```

**Problem**:
- No automatic cleanup on early return
- Exception safety not guaranteed
- Inconsistent with rest of codebase (which uses `std::vector`, `std::unique_ptr`)

**Fix**: Use `std::vector<uint8_t>` with RAII:
```cpp
std::vector<uint8_t> buffer(bufferSize);
PIP_ADAPTER_ADDRESSES addresses = 
    reinterpret_cast<PIP_ADAPTER_ADDRESSES>(buffer.data());

DWORD result = GetAdaptersAddresses(AF_UNSPEC, GAA_FLAG_INCLUDE_PREFIX, 
                                     nullptr, addresses, &bufferSize);
// Automatic cleanup when buffer goes out of scope
```

**Impact**: Memory leak risk on early returns, exception safety issues.

### 🟠 R14-H02: Thread Pool — Exception Swallowing

| Property | Value |
|----------|-------|
| **File** | `src/core/src/ncp_thread_pool.cpp` |
| **Lines** | 26-32 |
| **Severity** | 🟠 **HIGH** (CWE-755: Improper Exception Handling) |
| **Status** | ⬜ Open |

**Vulnerable Code**:
```cpp
void ThreadPool::ThreadPool(size_t num_threads) {
    for (size_t i = 0; i < num_threads; ++i) {
        workers_.emplace_back([this] {
            while (true) {
                std::function<void()> task;
                {
                    std::unique_lock<std::mutex> lock(queue_mutex_);
                    condition_.wait(lock, [this] {
                        return stop_ || !tasks_.empty();
                    });
                    if (stop_ && tasks_.empty()) {
                        return;
                    }
                    task = std::move(tasks_.front());
                    tasks_.pop();
                }
                ++active_;
                try {
                    task();
                } catch (...) {
                    --active_;
                    throw;  // ❌ CRITICAL: Rethrowing in worker thread!
                }
                --active_;
            }
        });
    }
}
```

**Problem**: `throw` in catch block causes `std::terminate()` in worker thread!

**Exploit Scenario**:
1. Task throws exception
2. Worker thread catches and rethrows
3. No outer handler in thread function
4. `std::terminate()` called — **entire process crashes**

**Fix**:
```cpp
try {
    task();
} catch (const std::exception& e) {
    --active_;
    // Log error but don't crash thread
    std::cerr << "[ThreadPool] Task exception: " << e.what() << std::endl;
    // Continue processing other tasks
} catch (...) {
    --active_;
    std::cerr << "[ThreadPool] Task unknown exception" << std::endl;
}
```

**Impact**: Single task exception crashes entire application.

### 🟠 R14-H03: OpenSSL Resource Management — Manual Cleanup

| Property | Value |
|----------|-------|
| **File** | `src/core/src/e2e.cpp` |
| **Lines** | 69-105 |
| **Severity** | 🟠 **HIGH** (CWE-763: Release of Invalid Pointer) |
| **Status** | ⬜ Open |

**Code Pattern**:
```cpp
static UniqueEVP_PKEY ec_p256_pkey_from_private_legacy(...) {
    EC_KEY* ec_key = EC_KEY_new_by_curve_name(NID_X9_62_prime256v1);
    if (!ec_key) throw std::runtime_error("EC_KEY_new_by_curve_name failed");

    UniqueBN priv_bn(BN_bin2bn(priv_raw, static_cast<int>(priv_len), nullptr));
    if (!priv_bn) {
        EC_KEY_free(ec_key);  // ✅ Manual cleanup
        throw std::runtime_error("BN_bin2bn failed");
    }

    if (EC_KEY_set_private_key(ec_key, priv_bn.get()) != 1) {
        EC_KEY_free(ec_key);  // ✅ Manual cleanup
        throw std::runtime_error("EC_KEY_set_private_key failed");
    }

    // ... more error paths with manual EC_KEY_free ...

    EVP_PKEY* pkey = EVP_PKEY_new();
    if (EVP_PKEY_set1_EC_KEY(pkey, ec_key) != 1) {
        EC_KEY_free(ec_key);  // ✅ Manual cleanup
        throw std::runtime_error("EVP_PKEY_set1_EC_KEY failed");
    }
    EC_KEY_free(ec_key);  // ✅ Manual cleanup
    return UniqueEVP_PKEY(pkey);
}
```

**Assessment**: ⚠️ **FRAGILE** — All error paths have manual cleanup, but pattern is error-prone.

**Better Pattern**: Use RAII from start:
```cpp
static UniqueEVP_PKEY ec_p256_pkey_from_private_legacy(...) {
    UniqueEC_KEY ec_key(EC_KEY_new_by_curve_name(NID_X9_62_prime256v1));
    if (!ec_key) throw std::runtime_error("EC_KEY_new_by_curve_name failed");

    UniqueBN priv_bn(BN_bin2bn(...));
    if (!priv_bn) throw std::runtime_error("BN_bin2bn failed");
    // No manual cleanup needed!

    if (EC_KEY_set_private_key(ec_key.get(), priv_bn.get()) != 1) {
        throw std::runtime_error("EC_KEY_set_private_key failed");
    }
    // Automatic cleanup on exception!
}
```

**Impact**: Future modifications may introduce memory leaks.

### 🟠 R14-H04: SecureMemory — Missing Bounds Check in Constructor

| Property | Value |
|----------|-------|
| **File** | `src/core/src/ncp_secure_memory.cpp` |
| **Lines** | 35-42 |
| **Severity** | 🟠 **HIGH** (CWE-122: Heap-based Buffer Overflow) |
| **Status** | ⬜ Open |

**Code**:
```cpp
SecureMemory::SecureMemory(const uint8_t* data, size_t size)
    : size_(size)
{
    if (size > 0 && data) {  // ✅ NULL check
        data_ = new uint8_t[size];
        std::memcpy(data_, data, size);  // ❌ No size validation
    }
}
```

**Issue**: No maximum size validation. Attacker could request gigabytes.

**Exploit Scenario**:
1. Attacker controls `size` parameter
2. Requests `size = 0xFFFFFFFF` (4GB)
3. `new uint8_t[size]` may succeed (throws `std::bad_alloc` on failure)
4. Memory exhaustion / DoS

**Fix**:
```cpp
static constexpr size_t MAX_SECURE_MEMORY = 1024 * 1024 * 100;  // 100MB

SecureMemory::SecureMemory(const uint8_t* data, size_t size)
    : size_(size)
{
    if (size > MAX_SECURE_MEMORY) {
        throw std::runtime_error("SecureMemory size exceeds maximum");
    }
    if (size > 0 && data) {
        data_ = new uint8_t[size];
        std::memcpy(data_, data, size);
    }
}
```
